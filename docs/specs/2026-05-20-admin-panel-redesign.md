# Admin Panel 重做 · 业务分析与设计

> 本文档面向产品/运营/管理层，**不讨论技术实现**。
> 写作目标：让一个第一次接触本项目的人，读完之后能讲清楚我们的数据底层、用户体验承诺、以及新 Admin Panel 长什么样。
>
> 起草日期：2026 年 5 月 20 日
> 起草依据：6 份独立调研（见末尾「调研依据」节）

---

## 一、为什么要重做

当前的 Admin Panel（`audit/audit.html`）是一份**只读快照**——只能看到爬虫抓了什么数字、覆盖率多少，却没办法把一个错误数据"标注为错"并"改成正确值"。下次重新抓数据时，错误的数据依然会被原样覆盖回来。

这次重做不只是改前端，而是借这个机会把底层数据模型 **整体推翻、重新梳理**。原因是这次调研发现：上一代爬虫产出的数据，在多个关键维度上**有结构性错误**，不是 bug 修一修就行，而是数据模型本身少了几个维度。本文档把"该有的维度"和"新面板该怎么承载这些维度"一次性讲清楚。

---

## 二、我们对用户的承诺（产品定位）

这个产品的核心价值是一句话：

> **准确告诉用户：哪些博物馆 pass 你真的能领，哪些不能领。如果不能领，告诉他为什么；如果是时间问题，告诉他什么时候可以。**

"准确"是关键。一个家庭兴冲冲跑去图书馆取 pass，被告知"你不符合资格"，体验是负的。所以**宁可漏推一条，也别推错一条**——这条原则贯穿后面所有数据模型和推荐逻辑的取舍。

---

## 三、底层需要哪些维度的数据

我们把数据按"实体"组织。一共 5 类实体，每类一节，配字段说明表。

### 3.1 景点 (Attraction)

景点是用户最终想去的目的地。一家博物馆、动物园、剧院或公园就是一个景点。

| 字段 | 业务含义 | 例子 |
|---|---|---|
| 名称 | 景点名 | Museum of Fine Arts |
| 地理位置 | 地址 + 经纬度 | 465 Huntington Ave, Boston / 42.339, -71.094 |
| 头图 | 一张代表性图片 | mfa.png |
| 开放时间 | 每周营业时间表 + 季节性闭馆 | 周一闭馆 / 7-12 月每天开 |
| 票价 | 按人群（成人/儿童/老人/学生/军人/教师/家庭）分别的官方价 | 成人 $27 / 老人 $25 / 学生 $10 |
| **预约要求** | 是否需要在景点官网先订时段才能进 | MFA 必须预约 / Museum of Science 必须预约 |
| **持卡人预约通道** | 持图书馆 pass 的人怎么订时段（4 种通道之一） | promo code 走通用结账 / 专属 SKU / 专属 URL / 仅图书馆代订 |
| **访客 residency 规则** | 景点本身对访客是否有居住地限制 | Harvard Museum of Science: 仅 MA 居民 / PEM: Salem 居民免费 |

后两项是这次调研新加的，前面的版本完全没有。详见调研发现章节。

### 3.2 图书馆 (Library)

图书馆是 pass 的供给方。一个图书馆系统就是一个 Library 实体。

| 字段 | 业务含义 | 例子 |
|---|---|---|
| 名称、城镇、地址、经纬度 | 基本元数据 | Wakefield Library / Wakefield 镇 |
| 所属联盟 | 见 3.3 节 | NOBLE |
| **办卡资格** | 谁能办这家馆的实体卡 | MA 居民 / 本镇居民 / 本镇居民或在本镇工作 / 联盟内人 / 不限 / 未知 |
| **取 pass 资格** | 谁能用这张卡领这家馆的 pass | 与办卡同 / MA 居民 / 仅本镇 / 仅本馆发卡的人 / 联盟内人 / 非居民可同日 walk-in / 未知 |
| 分馆列表 | 仅 BPL/Cambridge/Brookline 有 | BPL 有 16 个分馆做 pass 领取点 |
| 开放时间 | 馆的每周营业时间 | 周日闭馆 / 周二—周六 10:00-20:00 |

**关键点**：办卡资格和取 pass 资格是**两个独立的字段**。同一个馆，两者可能完全不同：

- Wilmington Library：办卡限本镇居民，但取 pass 对整个 MVLC 联盟开放
- Lexington Library：办卡对整个 Minuteman 联盟开放，但取 pass 仅限 Lexington 居民
- Wakefield Library：取 pass 分两档——本镇居民可提前预约，非居民只能当天 walk-in

### 3.3 联盟 (Network)

联盟是图书馆之间的合作组织。一张卡能不能"跨馆"使用，传统认知是看联盟，但实际情况比这复杂得多（见第四章调研发现 2）。

在本项目里，**联盟字段只承担两个有限作用**：

1. **划定爬虫和查询的硬边界**——调研确认没有任何跨联盟的阳性特例（例如：OCLN 的卡绝不可能在 NOBLE 馆借 pass），所以用户的卡可以用联盟字段过滤掉 75% 的查询。
2. **展示分组**——Admin Panel 和前端按联盟把馆分组折叠，让用户看 16 家 NOBLE 馆而不是看 59 家混在一起。

**联盟不再代表"内部全互通"**——联盟内有大量馆是"仅限本镇居民"的（例如 MVLC 36 馆里有 16 馆 town-only）。互通规则要逐馆看，不要凭联盟猜。

### 3.4 Pass（图书馆 × 景点 的合约）

Pass 是项目最复杂的一类实体。一行 Pass 代表"图书馆 A 给景点 B 的某种福利"。59 馆 × 平均 18 个合作景点 ≈ 1000+ 行 Pass。

每条 Pass 包含 4 层信息：

**第一层 · 领取方式**

| 类型 | 业务含义 |
|---|---|
| 电子邮件 (Digital / Email) | 图书馆把一个链接或 coupon 码发到你邮箱，你打印或截图，到景点出示 |
| 到馆自取 (Physical Coupon / Pickup) | 你需要本人去图书馆窗口取一张实体券，凭券去景点 |
| 自取并归还 (Physical Circulation / Pickup-and-return) | 拿的是一张可重复使用的实体 pass，用完要还回图书馆 |

**第二层 · 优惠内容（coupon）**

一条 Pass 下面挂多个"人群条款"——每个人群独立一条规则。

举例：Museum of Science 的某馆 pass 可能是这样：

| 人群 | 年龄 | 形式 | 数值 |
|---|---|---|---|
| 成人 | — | 半价 | 50% |
| 儿童 | 3 岁以上 | 减免固定金额 | $2 |
| 儿童 | 3 岁以下 | 免费 | — |

形式（form）枚举共 6 种：免费、按百分比折扣、固定减免金额、持卡入园价（不论原价多少，持卡就是 $X/人）、买一送一 (BOGO)、笼统折扣（只说"打折"没明确比例）。

容量（capacity）刻画"这张 pass 一次能用几个人"，单位可能是人、车（停车场 pass）、票（特定数量门票）、未指明。如果是"5 人家庭票"或"2 大 4 小一家"，统一记为总人数 5 或 6，原文保留在 source_phrases 字段供详情页展示。

**第三层 · 资格 override（少数情况才用）**

调研结果显示：约 95% 的 Pass 跟随该图书馆的"取 pass 资格"默认值；只有约 5% 的 Pass 有自己单独的资格规则。这种少数情况靠 `eligibility_override` 字段表达。最典型的例子是 **Andover Library 的州立公园 pass**——其他 Andover pass 对整个 MVLC 联盟开放，但州立公园 pass 仅限 Andover 镇居民，因为这条 pass 是 Andover 镇政府预算资助的。

**第四层 · 时间限制**

包括：黑名单日期（blackout，例如 12 月 25 日不能用）、按周限制（仅平日 / 仅周末）、季节性（夏季 only）、需提前 X 小时预约。这些是过滤条件，不参与推荐排序。

**注意**：库存（availability）属于 Pass 的动态字段，每天/每小时刷新，**不进入人工审计**。

### 3.5 审计层 (Audit Override)

这是新增的实体。审计层和爬虫层是**两个独立的数据层**。

| 字段 | 业务含义 |
|---|---|
| target | 指向某个具体字段（例如：mfa 景点 → 成人票价） |
| status | 已核 (verified_ok) / 已改 (corrected) / 仅备注 (noted) |
| corrected_value | 如果是"已改"，正确的值放这里 |
| note | 自由文本备注 |
| audited_at, audited_by | 谁审的、什么时候审的 |

构建产物时，审计层永远覆盖爬虫层。这就是为什么"即使爬虫长期会出错，我们也能稳定"——人工修正一次，永久生效，下次重抓不会冲掉。

## 四、六个调研发现（最关键的颠覆点）

本节用故事方式讲清楚：为什么这次必须推翻重做，而不是修修补补。

### 发现 1：办卡资格 ≠ 取 pass 资格

我们采样 10 家图书馆，验证了一个假设：**这两件事是独立的两件事**，而且方向是双向的。

- Wilmington Library：办卡严（仅本镇居民），但取 pass 松（整个 MVLC 联盟）
- Lexington Library：办卡松（整个 Minuteman 联盟），但取 pass 严（仅 Lexington 本镇）
- Wakefield Library：取 pass 还分两档——本镇人可提前预约，非居民只能当天到现场 walk-in

**对模型的影响**：每个图书馆必须分开记录这两个字段，不能用一个 `eligibility` 字段替代。

### 发现 2：联盟内不一定通用

传统认知是"办了 NOBLE 卡 = 所有 NOBLE 馆都能借 pass"。**这是错的**。调研 4 个联盟的官方政策页和成员馆政策页后确认：

- **没有任何联盟发布过统一的 pass 互借规则**。每个馆自己定。
- 最接近"全互通"的 MVLC 联盟，36 个成员里仍有 16 个明确"仅本镇居民"。
- NOBLE / Minuteman / OCLN 联盟里大多数馆是"仅本镇居民"。开放给联盟内任意持卡人是少数特例，例如 Cambridge、Wellesley（同日 walk-in）、Weymouth（2023 年 7 月后开放）。
- **没有发现任何跨联盟阳性特例**——OCLN 卡绝对不可能在 NOBLE 馆借 pass。

**对模型的影响**：联盟字段仅作标签和爬虫边界使用，绝不能编程为"互通默认值"。互通信息必须逐馆人工录入或抓取。

### 发现 3：BPL eCard 不能借 pass

任意 MA 居民可以在网上申请 BPL 的 eCard，但 BPL 官网原话：**"eCards do not allow you to check out physical items such as museum passes."** 要借 pass 必须本人到 BPL 任一分馆面亲自升级到实体卡。

这是用户最容易踩的坑——很多家庭以为网上申请了 BPL eCard 就能用 pass，到馆才发现办不了。

**对模型的影响**：
- 我们的"图书馆卡"概念默认指实体卡，eCard 不进模型。
- 前端添加图书馆卡时，若是 BPL，需要显著提示"必须是实体卡，eCard 不能借 pass"。

### 发现 4：景点 timed-entry 独立于 pass

我们以为"用 pass = 进得了门"。**调研 5 家热门景点后发现并非如此**：

- MFA、MOS、ICA、Boston Children's Museum、ISGM、PEM 等景点要求**所有访客**必须在景点官网提前订时段才能进——持图书馆 pass 也不例外。
- 持 pass 的人不能跳过这一步。他必须先从图书馆拿到 coupon 码或 pass，再去景点官网用那个码订时段。
- 不同景点的"持卡人订时段通道"形式不一样：通用结账输入 promo code（MFA、PEM、ISGM）、专属 SKU 单独选（BCM）、专属 URL（MoS）、或景点根本不开放在线订、只能图书馆代订（极少数）。
- 最严格的是 ISGM：必须提前至少 48 小时订。

**对模型的影响**：景点实体加 "预约要求 + 持卡人通道" 字段。所有 timed-entry 景点的卡片必须显示**两步指引**（先领码 → 再去官网订时段），不然用户必跑空。

### 发现 5：coupon 形态有几个藏雷

抽样了 60 条 pass 的原始抽取 JSON，发现 3 个会让简单 schema 出错的问题：

1. **买一送一 (BOGO) 被错抹成"50% 折扣"**——例如 Boston Harbor Islands 渡轮 pass，原文说 "buy one get one free"，被 LLM 抽成"50% off"。**单独一个人去，实际优惠是 0%，不是 50%**。前端如果显示"50% off"是对单人用户撒谎。所以 form 枚举必须新增"BOGO"。

2. **黑名单日期 (blackout) 硬编码了 2026 年**——意味着进 2027 年这些黑名单全部失效，需要重新抓。改用相对日历（只存月日，不存年份）。

3. **资金来源不等于资格限制**——例如某 Wilmington pass 写"由 Wilmington Council for the Arts 资助"，被错认为"仅 Wilmington 居民可领"。**资助方 ≠ 资格限制**。这是一种 LLM 抽取的常见误判，需要人工审计层兜底。

### 发现 6：景点也有 residency 规则

这是 A6 调研补刀的发现——之前我们只考虑"图书馆对人有要求"和"pass 对人有要求"，**没考虑景点自己对访客的要求**。但事实是：

- Harvard Museum of Science & Culture：仅 MA 居民可入
- PEM (Peabody Essex Museum)：Salem 镇居民免费
- JFK 图书馆 & 博物馆：对任意 MA 图书馆卡持有人开放

**对模型的影响**：景点实体加"访客 residency 规则"字段。这是一个独立于图书馆 pass 的过滤层。

### 发现 7：BPL 分馆模式

BPL 有 25 个分馆，曾考虑把每个分馆当独立图书馆建模（library 表会 5 倍膨胀）。调研后决定：

- **BPL 作为一个图书馆实体**，分馆作为 `branches[]` 子列表挂在上面。
- 每条 BPL pass 上加 `available_at_branches` 字段，列出哪些分馆有库存（默认"全部"）。
- Cambridge（7 分馆）、Brookline（3 分馆）同样模式。Assabet 平台的 52 馆和 museumkey 的 2 馆都是单建筑，不需要这套机制。
- 取 pass 的"分馆开馆时间"成为漏斗的一个独立检查层（见下一章）。

## 五、用户漏斗：从输入到结果的 10 个过滤层

用户进入产品后输入：①持卡列表，②home_resident（居住地，决定办卡和取 pass 资格），③想去的景点和日期。

如果没拿到 pass，系统必须**清晰告诉用户卡在哪一层**，而不只是说"暂无优惠"。10 层从永久性到时效性排列：

| 层 | 检测内容 | 用户看到的解释 | 未来何时可用？ |
|---|---|---|---|
| L1 | 我是否持有该图书馆/联盟的实体卡？ | "你没有 NOBLE 网络的卡，这家 pass 不适用" | 永久——除非办卡 |
| L2 | 我的 home_resident 是否满足该馆办卡资格？ | "Lexington 仅本镇居民可办卡，你住在 Reading" | 永久——除非搬家 |
| L3 | 我的 home_resident 是否满足该馆取 pass 资格？（含 pass 自身的 override） | "Andover 州立公园 pass 仅本镇居民可领，你住在别处" | 永久 |
| L4 | 我的 home_resident 是否满足景点本身的 residency 规则？ | "Harvard Museum 仅 MA 居民可入，你不是 MA 居民" | 永久 |
| L5 | 景点在查询日是否开放？ | "MFA 周一闭馆" | 下一个开放日 |
| L6 | 图书馆在查询日是否开馆？（仅需到馆取的 pass） | "Wakefield Library 周日闭馆，无法领取" | 下一个开馆日 |
| L7 | 取 pass 的分馆在查询日是否开馆？（仅 BPL/Cambridge/Brookline） | "BPL Brighton 分馆周三关门" | 下一个开馆日 |
| L8 | 是否撞上 blackout / 仅平日 / 季节性闭区？ | "这张 pass 7 月 4 日 blackout" | 下一个合规日 |
| L9 | 是否满足提前预约时长？ | "ISGM 需提前 48 小时订，今天太迟" | 距今 +48 小时后 |
| L10 | 当天 pass 库存是否还有？ | "今日已订满" | 下一个 available 日 |

L1—L4 是**永久层**，时间维度无法解锁；L5—L10 是**时效层**，能告诉用户"那么什么时候可以？"。

**unknown 资格的处理**：在 L2、L3、L4 出现 unknown 时，按"通过"处理（让用户能看到推荐），但在结果卡片上加 ⚠️ 角标"资格未确认"，并让用户后续在 Admin Panel 人工修正。

## 六、Pass 推荐逻辑

用户进入一个景点详情页时，系统最多展示 **4 条 pass 领取推荐**。

### 6.1 类别配额（最多 4 条 = 1 + 3）

| 类别 | 优先级 | 上限 | 类内规则 |
|---|---|---|---|
| Email pass | 最高 | 1 条 | 任何馆的 Email pass 对同一景点效果等同——按 (景点, coupon 内容哈希) 去重，只取最优一条 |
| 自取 pass (Pickup) | 中 | 3 条 | 按优惠力度 + 距离打分 |
| 自取并归还 (Pickup-and-return) | 低 | 填补剩余位 | 同上 |

### 6.2 打分规则

每条候选 pass 计算一个分数，分高的排前：

- 优惠力度权重（free > 高折扣 > 固定减免 > 持卡入园价 > 笼统折扣 > BOGO 对单人）
- 距离权重（仅 pickup 类——取 pass 的图书馆离用户家近的加分）
- 资格不明 (unknown) 排末（不剔除）

### 6.3 不参与打分的事

时间限制（blackout / 提前预约 / 仅平日）**不打分**，只做硬过滤——撞上就直接不显示。原因：这些字段本身就不稳定（图书馆和景点的政策日历经常变），用它们扣分会让推荐预测偏差更大。

## 七、Admin Panel 新设计

Admin Panel 不是"被动等审"的待办队列，而是一个**带纠错能力的多维查询透视台**。打开它的人有可能在做这几件事之一：

- "我想看我手上的 5 张卡，能在哪些景点拿到优惠"
- "我想看 BPL 的 16 个分馆有什么差异"
- "我想审 MFA 的 12 条 pass，看哪个馆的优惠力度数据是错的"
- "我想看 Wakefield 这家馆的所有政策字段是否完整"

不同场景关心不同字段，所以面板必须支持**视角切换**——按场景收起无关字段，避免大宽表 1000 行 × 20 列把人晃晕。

### 7.1 顶部：四个视角 (Lens)

| 视角 | 主视图行单位 | 默认显示 | 默认折叠 |
|---|---|---|---|
| A · 卡覆盖查询 | 一行 = 一个 Pass | 图书馆名、景点名、我合规吗、一句话优惠摘要、当日库存 | 多人群拆分、blackout 细节、原文证据 |
| B · 资格政策审查 | 一行 = 一个图书馆 | 联盟、办卡资格、取 pass 资格、card_page 链接 | 优惠细节、库存 |
| C · 优惠细节准确性 | 一行 = 一个 Pass | 全展开人群条款、form、value、容量 | 营业时间、经纬度 |
| D · 分馆与景点预约 | 一行 = 一个景点或分馆 | 访客 residency、预约要求、持卡人通道、分馆列表 | 优惠 |

### 7.2 中部：大宽表（数据透视风格）

- **按联盟分组折叠**：默认列头显示 "NOBLE · 16 家 [+]"，点开展开。参考 backup/ 那版的联盟分组视觉，比扁平 59 行更像图书馆世界的真实样子。
- **行筛选**：home_resident + 持卡列表 + 日期可以在顶栏输入，先把 1000+ 行筛到合理量级再渲染。
- **单元格交互**：鼠标 hover 时右上角出现三个按钮——✅ 已核 / ✏️ 改值 / 📝 备注。任何被人工改过的单元格左上角带一个小角标，点开看完整 audit log。

### 7.3 右上：漏斗模拟器

一个独立小工具。输入"我是某 town 居民 + 持某几张卡 + 某一天想去某景点"，输出每个景点卡在 L1—L10 的哪一层；对时效层（L5—L10）给出"下一个可用日"。

这个模拟器既是 Admin 自测产品逻辑的工具，也是给运营/老板演示产品准确性的工具。

### 7.4 底部：纠错日志查询

按字段、按审计人、按日期范围、按状态反查所有 audit 写入。

### 7.5 明确不做的事

- 不做"待审队列"——不催完成
- 不做"完成度 %" 仪表盘——会诱导面子工程
- 不审库存——库存是纯动态层，每次刷新都变，审了也没意义
- 不直接编辑爬虫层 raw 数据——所有修改写入审计层 (audit overrides)，构建时合并
