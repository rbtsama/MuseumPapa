Passes优惠券 · 7 个角度看 1026 条 pass

每张 pass = 1 家图书馆 × 1 家景点 × 1 份优惠条件。本页从 7 个维度告诉你这 1026 条 pass 是什么、分布怎样、对产品意味着什么。下方每节按 What it is · 分布 · What it means 三段写,跳过 ID 字段——明细查 Schema,边界案例查 Data Quality。

1 · What is a Pass什么是一张 Pass

What it is · 这是什么: 每条 pass = 一家图书馆 × 一家景点 × 一份优惠条件。同一家景点可能在多个图书馆出现,但每条 pass 的取卡方式、折扣力度、能用几人都可能不同——所以不会被去重成一条。

Passes per library (mean)每馆平均 pass 数17.4—
Libraries per attraction (mean)每景点平均覆盖馆数10.6—
What it means · 含义:
  • 分布是稠密的——59 个馆里 100% 都至少有 pass,97 个景点平均每个被 10.6 个馆覆盖。对持单卡用户来说,你能用的 pass 平均有 17 张;对持多卡(运营方 5 张)用户,数量乘到 ~85 张。
  • 同景点跨馆条件不同是常态——比如 Salem Witch Museum 在 12 个馆,实际折后价有 4 档($13.25 / $13.5 / $13.75 / $14)。Pass 不去重正是为了让多卡用户能横向对比。

2 · Acquisition path (Platform)取卡途径 · 数据来自哪 3 个系统

What it is · 这是什么: 数据从 3 个图书馆系统抓回来。哪个馆走哪个平台不是我们选的——是图书馆自己买的产品。这一点决定了 能不能拿到实时库存。

What it means · 含义:
  • Assabet Interactive · 全 MA 公共图书馆通用的 pass 预订平台,覆盖 52 家小型馆。HTML 暴露日历,我们能解析出 available / limited / booked 三态。有实时库存。
  • LibCal (Springshare) · 中大型馆产品。BPL、Cambridge、Brookline、Braintree、Milton 用。日历通过 institution 端点取,3 种状态合并为 available/booked。有实时库存。
  • MuseumKey · 商业 SaaS,要图书馆卡登录才能看 pass。Cohasset 和 Hingham 用。我们只能拿到 catalog(景点列表、benefit 文本),没有实时库存——用户看到这两个馆的 pass 时,日历位置是空白,要打电话或上馆里现场查。

3 · How the user picks it up怎么取 · digital / physical-coupon / physical-circ

What it is · 这是什么: pass_type 描述用户怎么把券拿到手——纯线上、还是要去馆里。这个差异直接影响产品体验:digital 是抢菜模式,physical 是有提前量但更稳。

digital · Email-delivered邮件即时收券码 / PDF███████████████████████58657%
physical-coupon · Pick up paper coupon去馆领纸券██████████26726%
physical-circ · Borrow physical pass借实体卡,看完归还███████17317%
What it means · 含义:
  • 57% 是 digital——填表后邮件秒到,可以当天预约当天用。前端配实时日历后,这部分是体验最好的 pass。
  • 26% 是 physical-coupon(纸券)——一般要 24-48h 才能拿到,适合提前 1-2 天规划。
  • 17% 是 physical-circ(实体卡)——占用图书馆的循环库存,看完要还,适合提前 3-7 天约。在我们的库存逻辑里,这种 pass 看完没还回来之前,日历就是 'booked'。

4 · What discount form给什么折扣 · 5 类 form 分布

What it is · 这是什么: 每条 pass 的 audience_policies 列表里有 1-3 条折扣描述(共 1317 条),每条用 form 字段标明折扣类型。前端就是根据这 5 类决定怎么把优惠串成一句话给用户。

Per-person negotiated price馆员协议价(如 $10/人)███████████████48737%
Percent off百分比折扣(如 50% off)█████████████43833%
FREE完全免费████████27221%
Vague discount (no number)笼统折扣 · 原文无具体数值███907%
Dollar off固定金额减免(如 $5 off)█302%
What it means · 含义:
  • 前两类合计 71%(per-person 37% + percent-off 34%)——这部分我们能直接给出'每人多少钱'或'省百分之几',用户心算对比原价,决策成本最低。
  • FREE 20%——pass 类型里最强的优惠形态,值得在 UI 上突出。
  • vague discount 6%(90 条)——图书馆原文只说 'discounted admission' 或 'family rate',上游本来就没数字。属表达问题不是抽取 bug;前端只能显示'有折扣',无法说省多少。
  • $N off 仅 2%——固定减额罕见,通常出现在低价景点(Concord Museum $5 off 这种)。

5 · How many people fit能用几人 · capacity

What it is · 这是什么: capacity 描述一次能带几个人。两个字段:kind(算人 / 算车 / 算票 / 未明示)和 n(数量)。用户看到 'Up to 4' 那个数字就是从这来的。

Per vehicle (n=1)按车 · 全部默认 1 辆(MA State Parks 等户外公园场景)██596%
Tickets按张 · 剧院/演出常见(2-4 张戏票)██505%
Unspecified未明示 · 上游原文没说人数██606%
What it means · 含义:
  • 84% 是限人数(857 张),其中 'Up to 4' 是远超过半的众数(379 条,44%)——对应典型的两大一小家庭出行规模。'Up to 6' 排第二(147 条)。
  • vehicle 59 张全部是 1 辆车,这是个产品默认值(原文从不写'1');建议 UI 上显式渲染成 'Per vehicle (1 car)' 避免用户误以为'按车每辆都行'。
  • unspecified 60 张——经子代理逐条复核(见 Data Quality §unspec),35 条是 by-audience(各受众自己带 count,全局 n 无意义),9 条是 family-tier 整张通票,16 条是上游原文真模糊。这 60 条不是抽取 bug,是 schema 设计选择 / 上游限制。

6 · Audience split人群拆分 · 1 / 2 / 3 类各享不同待遇

What it is · 这是什么: 一张 pass 可以把人群拆成 1-5 类,每类各享不同折扣。audience_policies 列表的长度就是这里的'类数'。

2 audience(s)2 类受众 · 典型: Adult $X/人 + Child <16 free██████████26726%
3 audience(s)3 类受众 · 罕见 · Adult / Child / Senior 各自不同价█101%
5 audience(s)5 类受众 · 极个别 outlier · 值得人工 review█10%
What it means · 含义:
  • 74% 是单类(一刀切),折扣不区分大人小孩。这部分 pass 信息简单,UI 一句话就能说完。
  • 26% 是双类——这是产品最有信号的形态:能告诉用户'大人 $X、孩子免费',对带娃出行决策的价值远大于'大家都 50%'。前端值得突出渲染双类 pass。
  • 3+ 类极罕见(<1%),这种 pass 通常是 Adult / Child / Senior 三档,或者带上 Student / Educator / Military 的特殊优惠。

7 · Usage restrictions使用限制 · blackout / weekday / seasonal / reservation

What it is · 这是什么: restrictions 字段记录这张 pass 哪天不能用 / 要不要预约。全部 1026 张中有 69 张带某种限制(其余无限制,任意可用日都成立)。前端把这些限制渲染成 ⚠ 角标,但不会因此把 pass 排除——只是提醒。

What it means · 含义:
  • 69 / 1026 张(6.7%)带某种限制——比例不高,但都是日历层面的硬约束,前端不能不展示。
  • 季节性 40 张占多数,典型场景:Boston Harbor Islands、Crane Beach 这种夏/秋限定的户外目的地。winter 时用户能看到 pass 但日历全灰。
  • blackout 29 张——通常是图书馆和景点谈下'特定日子不让用'的协议(节假日、特展开幕)。前端把这些日期日历格变 ⚠,点开告诉用户原因。
  • reservation_required 在这版数据里是 0,但 schema 留着——多见于热门博物馆,后续抓到再填。
